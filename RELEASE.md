# Release process

Releases fail closed. `npm run release:validate` rejects placeholder values for the deliberately public release constants — canonical domain, DNS release-key owner, maintainer roster, and Minisign public key — requires `git verify-commit HEAD` to succeed against the maintainer's local trust configuration, and runs the repository's automated local source, build, reproducibility, release, and browser gates. The hosted workflow independently verifies the exact commit signature once through GitHub in its protected preflight; fresh build containers rely on that successful preflight and do not require an unprovisioned local keyring. These commands do not collect or validate every external fact. The manual and hosted evidence gates below (protected `main`, a signed clean commit, rendered Cloudflare state, physical-browser evidence, independent review, hosted dual-build and attestation evidence, signing-ceremony evidence, and production-rehearsal evidence) remain mandatory for every release.

## Candidate gate

1. Require protected `main`, a clean signed commit, exact tool/action/container pins, matching version metadata, exactly one dated changelog heading for that version, current PSL/IANA snapshots, reviewed Unicode data, passing CI/security/browser/offline/no-egress gates, and successful `npm run release:validate`. Accessibility verification today consists of the hand-written assertions in `tests/browser/` (forced-colors rendering, the skip link, and view focus handoff) plus manual keyboard, screen-reader, and reduced-motion checks; no automated axe-style scanner runs in CI.
2. Build twice in isolated pinned environments and compare every unsigned byte. Verify decoder package integrity and the locked reader WASM SHA-256 `6a858c01e076bab3a1bd413e4f2cf5e5e45f819a0d9441d83c66993bc48ed38f`.
3. Produce the normalized source/dist archives, dist-files manifest, archive manifest, CycloneDX SBOM, license report, and version changelog. Verify the closed artifact contract, headers, MIME, cache, CSP, release marker, 2 MiB precache limit, and absence of source maps/absolute paths.
4. The dispatch-only workflow creates GitHub build/SBOM attestations tied to the unsigned candidate digests. Independently read back and verify the subjects, digests, repository, commit, and protected workflow identity; that post-upload attestation verification is a manual gate today. Attestations do not replace Minisign.
5. The automated workflow stops after uploading a byte-identical approved unsigned candidate. Follow `SIGNING.md` offline, verify the complete signed set independently, upload draft artifacts and signatures, read them back, and reverify before any deployment or publication. Signing, signed-set finalization, upload readback, and publication are manual gates today.

The automated deterministic base set is `qrwarden-X.Y.Z-source.tar.gz`, `-dist.tar.gz`, `-dist-files.sha256`, `-archive.sha256`, `-sbom.cdx.json`, `-licenses.txt`, and `-changelog.md`. The source archive, dist archive, dist-files manifest, and archive manifest require `.minisig` signatures. The repository defines transition/recovery input schemas and validates operator input, but deterministic statement generation, candidate inclusion, and signature-set integration are still pending. Do not claim or publish a transition/recovery release until that tooling is implemented, reviewed, and exercised end to end.

## Deployment state machine

Follow `docs/DEPLOY_CLOUDFLARE.md` for the exact operator sequence, asset-bearing upload configs, asset-free trigger configs, machine-readable evidence, persistent Access preview guards, live byte/header verification, exact domain-ID detachment, rollback, and cleanup. `wrangler versions upload` and `wrangler triggers deploy` are separate operations; never substitute `wrangler deploy`, which creates a version and immediately sends it to production.

Capture and byte-verify the single live 100-percent OLD version against the prior signed release (or v1 bootstrap baseline). Upload only a safely extracted, remanifested copy of the signed dist archive with repository-local Wrangler 4.111.0; capture exactly one NEW version ID and preview URL. Open the Access-protected preview window, create an explicit OLD 100% / NEW 0% deployment, and smoke-test NEW only through the exact version override.

The first public deployment currently has no independently verified OLD artifact. Treat it as a signed beta or release rehearsal: verify it through a route-less Access-protected preview, deploy it while the canonical domain remains detached, and attach the domain only after preview success. Its failure rollback is Workers Domains API deletion of the captured immutable domain ID followed by exact absence readback. A later release may use that retained signed beta as OLD and complete the stable-v1 rollback drill; the initial launch itself must not be described as satisfying that gate.

Recheck OLD immediately before promotion. Promote NEW atomically to 100%, poll until exact, and run fresh-profile, installed/offline, live header/body, zero-egress, and update tests. On failure within the controlled window, roll back only to the still-independently-verified OLD. Always remove the release-specific Access policy, service-token activity, override headers, and preview URLs while preserving the persistent deny-by-default Access guards; arm and test the watchdog. A service worker already received by clients requires a forward corrective release even after infrastructure rollback.

Only after live verification succeeds should maintainers publish the immutable GitHub release and tag. Existing releases, assets, signatures, and tags are never replaced or deleted. Store operational version/deployment/Access IDs outside deterministic artifacts and update the rollback target.

## Stable v1 gate

Stable v1 additionally requires name and domain clearance, independent security review, automated axe-style accessibility scanning wired into CI, physical current and previous platform testing with exact builds, reproducibility proof, signing ceremony, rollback and watchdog drills, permission and lifecycle tests, a 20-minute camera soak, 100 sequential image scans, and all documented acceptance gates on the exact signed candidate.
